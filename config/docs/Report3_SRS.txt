Software Requirement Specification


Product Overview
The Bengo Platform is an online system designed to seamlessly connect passengers with transportation providers for mass transit solutions. Key features include intelligent route and vehicle recommendations, detailed driver and fleet profiles, real-time in-app chat, subscription management for businesses, and advanced trip tracking tools. The platform aims to empower transportation partners with powerful fleet management and dispatch tools, while providing passengers with a smooth, efficient, and reliable commuting experience. With integrated notifications, multiple payment gateways, and AI-based ride-matching and route optimization, Bengo is designed to streamline mass transportation processes, ensuring high user satisfaction and scalability for metropolitan and regional use.

Actors


#

Actor

Description

1

Guest
A user who can access the platform without logging in. Guests can view the landing page, perform basic searches for available shipments, and learn about the platform's features. They cannot place bids, accept shipments, save shipments, or interact with shippers until they create an account.



2

Driver
A user who has created an account to find shipment loads. They can bid on shipments, manage their profile (company information, vehicles), upload relevant documents (insurance, vehicle inspection), and track shipment progress. They can also save favorite shipments, chat with shippers, and confirm trip completion.

3

Administrator
The administrator oversees the entire system and ensures smooth operation. They have full control over user accounts (ban/activate/delete), approve or reject shipment postings (if moderation is required), resolve violation reports, manage subscription plans (if any), and monitor system analytics and financial transactions.

Authentication


Registration


Function trigger: The user clicks on the “Register” link on the Login page or accesses the Register page with a link.


Function description: Customers, drivers or administrators can create an account on the Bengo system to use the service. The registration process requires basic information and OTP verification via email.


Screen layout:




Function details:


Data: Include Full Name, Email, Password, Phone, Role


Validation:


Email must be in the correct format and not already in the system.


Phone number must be in the correct format and not already in the system.


Password does not require special security rules (can contain letters, numbers, lowercase characters).


OTP verification code must be correct and valid.


Business rule: BR-01, BR-02, BR-03


Functionality:


In Normal Cases: If all information is valid, the system saves the account, sends OTP code to email, displays the message “Registration successful! Please check your email to verify your account.”


In Abnormal Cases: 


Email or phone number already exists → displays error message “Email or phone number already in use.”


Invalid or expired OTP → displays error message “OTP code is invalid or expired.”





Login




Function trigger: The user presses “Login” on the main page or from any page that requires authentication.


Function description: Users log in to the system using their registered email and password.



Screen layout:

































Function details:


Data: Include Email or Phone and Password


Validation:


Email/Phone number must exist in the system.


Password must match the information stored in the database.


Account must have successfully verified email (isEmailVerified = true).


Business rule: BR-04, BR-05


Functionality:


In Normal Cases: If the information is valid, the system authenticates and issues a JWT token to the user, then redirects to the home page or dashboard.








Forgot password



In Abnormal Cases:


Incorrect email or password → “Incorrect email or password.”


Unverified account → “Please verify email before logging in.”


Locked account → “Account temporarily locked. Please try again later.”


Function trigger: Users click &quot;Forgot Password&quot; on the login page when they forget their password.


Function description: The system allows users to reset their password via OTP sent to the registered email.


Screen layout:




Function details:


Data: Include email, otp, new password, confirm new password


Validation:
Email must be valid and exist in the system.
OTP must be correct and valid.
New password and confirmation must match.


Business rule: BR-06,BR-07


Functionality:


In Normal Cases: The system sends a verification OTP via email. When the user enters the correct OTP, the password is updated and a message “Password reset successful” is displayed.


In Abnormal Cases: 


Invalid or expired OTP → “OTP code is invalid or expired.”


Email does not exist → “Email is not registered in the system.”



User Account Management


View List User Accounts


Function trigger: The admin clicks on the &quot;User Management&quot; tab or selects &quot;View All Users&quot; from the admin dashboard.


Function description: The admin can view a paginated and searchable list of all user accounts registered in the Bengo system. The list displays basic user information and supports filtering by role, keyword, and status.


Screen layout:



Function details:






Data: User ID, Name, Email, Phone,Role, Email verification status,
Account Status, Created date, Last login date


Validation:


The system verifies admin privileges before displaying user data.


Only authenticated Admin accounts can access this feature.


Business rule: BR-10, BR-11


Functionality:


In Normal Cases: 


The system retrieves the user list with pagination (default 20 users per page, max 100).


If filters (e.g. role, search) are provided, the results are narrowed accordingly.


The admin can navigate between pages and view summary metadata (page, limit, total pages).


In Abnormal Cases: If the database connection fails → message: &quot;Unable to retrieve user accounts. Please try again later.&quot;. If no users match filters → &quot;No users match your search criteria.&quot;


View List Drivers


Function trigger: The admin selects the &quot;Driver List&quot; option from the sidebar or clicks the &quot;Drivers&quot; tab in the User Management section.


Function description: Allows the admin to view and manage all drivers in the system, including their linked user info, vehicle, and registration status.


Screen layout:



Function details:


Data: Driver ID, User info (name, email, phone), Status, Rating, Total trips, Income balance, Registration date


Validation:


Only Admin users can access this list.


The system ensures proper role-based authorization.


Business rule: BR-12


Functionality:


In Normal Cases: The system retrieves drivers with pagination, supports search by name/email/phone, and filters by status.


In Abnormal Cases: 


If no driver matches → &quot;No drivers found.&quot;


If DB fails → &quot;Unable to retrieve driver list.&quot;


View Driver Detail


Function trigger: The admin clicks on a specific driver in the list.


Function description: Shows full driver profile, vehicle info, recent transactions, and earnings summary.


Screen layout:



Function details:


Data: Driver info (status, rating, serviceAreas, avatarUrl),Linked user info: (name, email, phone, address), Vehicle list (if available), Order stats (Created, Completed, Cancelled), Transaction list (latest 10), Total earnings (netAmount, fee).


Validation:


Verify driver exists.


Only admin authorized accounts can view detailed data.


Business rule: BR-13


Functionality:


In Normal Cases: The system aggregates data from multiple collections (Driver, User, Order, DriverTransaction) and returns combined info.


In Abnormal Cases: 


&quot;Không tìm thấy tài xế&quot; if not found.


&quot;Lỗi lấy thông tin tài xế&quot; if query fails.

Driver Withdrawal


Function trigger: The driver clicks the &quot;Withdrawal&quot; tab in the Driver Dashboard or selects &quot;Request withdrawal&quot;.


Function description: The driver can submit a payout request from their available balance and view a paginated history of withdrawal requests with statuses and details. Drivers may cancel a pending request before it is processed.


Screen layout:



Function details:






Data: Amount (currency, VND), Payout method: Bank transfer, Bank name, Account holder name, Account number, Note (optional).


Validation:


Driver must be authenticated and have an Active driver profile.


Amount &gt;= Minimum withdrawal amount; Amount &lt;= Withdrawable balance.


Bank info required and valid format.


Optional: Daily request limit not exceeded.


Business rule:



BR-WD-01: Fee = configured fixed or percentage; Net = Amount − Fee.


BR-WD-02: When request is Pending/Processing, the requested amount is held and excluded from available balance.


BR-WD-03: Driver can cancel only while status = Pending.


BR-WD-04: System logs all status changes and notifies driver in real time.


BR-WD-05: Admin processes and marks requests; driver is read-only for processed items.


Functionality:


In Normal Cases: 


Submit withdrawal:


System validates driver status, balance, and form fields.


System creates a request with status = Pending, computes fee/net, updates held balance, and returns success.


Driver sees success toast and the new record at the top of history.


View balance and history:


System shows current balance and withdrawable amount.


History is retrieved with pagination (default 10 per page, max 50), sortable by date, filterable by status/date range.


View detail:


Driver opens a request to see full metadata, timeline, and admin notes.


Cancel request:


If status = Pending, system cancels the request, releases held amount, and updates history.


In Abnormal Cases:


Insufficient balance or below minimum → “Amount exceeds available balance” or “Below minimum”.


Invalid bank info → field-level errors.


Daily limit exceeded → “Daily withdrawal limit reached; try again tomorrow.”


Backend error → “Unable to create withdrawal request. Please try again later.”


Cancelling non-pending → “Only pending requests can be cancelled.”





Driver Revenue Reports &amp; Receipts


Function trigger: Driver opens the “Revenue” tab or selects “Export report” / “View receipt”.


Function description: Drivers view earnings over a time range, filter/search transactions, open order receipts, and export reports (CSV/PDF).


Screen layout:



Function details:






Data: 






Summary: totalGross, totalFee, totalNet, tripCount.






Transactions[]: { id, orderId, createdAt, type, status, gross, fee, net, note }.






Receipt: order info, distance/weight breakdown, surcharges, promotions, platform fee, driver net, payment method, reference code.


Validation:


Authenticated driver with Active status.


Valid date range (e.g., max 92 days).


page and limit within bounds.


Business rule:



BR-RV-01: net = gross − fee (rounded per config).


BR-RV-02: Only Settled counts toward withdrawable balance.


BR-RV-03: Adjustments may be positive or negative; included in net.


BR-RV-04: Receipts are visible only for the driver who fulfilled the order.


BR-RV-05: Large exports run as a background job; a download link is sent via notification.


Functionality:


In Normal Cases: 


View summary based on current filters.


Browse history with pagination and sorting.


Open a receipt; allow print/download PDF.


Export CSV/PDF for current filters; show “generating file” when queued.


In Abnormal Cases:


Range too wide → “Please narrow your date range.”


No data → empty state.


Server error/timeout → “Unable to load revenue. Please try again later.”


Accessing another driver’s receipt → 403.
